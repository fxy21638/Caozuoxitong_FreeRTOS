## 一、应用需求分析

### 1.1 项目背景与应用场景

工业物联网与智能传感领域广泛采用RS485差分总线实现多节点远距离数据采集与设备控制。本项目模拟一套典型的工业传感采集场景：部署1台管理端主机与2台采集前端从机，通过单条RS485双绞线互连，实现温度/湿度环境参数与设备姿态角的集中采集、远程显示与设备控制。系统适用于小型工业监控网、环境监测站、智能实验室数据采集等场景。

### 1.2 功能需求

**（1）多节点分级采集**
- 采集前端1（地址0x02）：每5秒采集一次温湿度数据，本地存储最近10组记录，通过TFT触摸屏实时显示当前数值与历史趋势；
- 采集前端2（地址0x03）：每2秒采集一次MPU6050 DMP解算姿态角（Pitch/Roll/Yaw），本地存储最近10组记录，LCD实时显示当前角度；
- 管理端（地址0x01）：每10秒轮询一次两台前端，依次读取温湿度和姿态角，本地分配两个存储区各保存最近10组远端数据。

**（2）RS485总线通信**
- 采用自定义定长/变长帧协议，帧结构为：帧头0xAA + 地址字节 + 数据长度 + 消息类型 + 传感数据 + CRC校验字节 + 帧尾0x55；
- 消息类型分为：0x01温湿度读取/应答、0x02姿态角读取/应答、0x03 LED控制（0x01点亮/0x00熄灭）；
- 通信容错机制：单帧接收超时100ms直接丢弃并重置状态机；采集前端仅解析匹配自身地址的帧；管理端500ms内未收到应答自动重发请求帧。

**（3）人机交互与存储**
- 三台设备均搭载TFT LCD触摸屏GUI界面，实时展示传感数据、通信状态、在线指示；
- 管理端GUI采用左右卡片式布局（温湿度卡片+姿态角卡片），历史记录区显示最近6条记录，状态栏展示更新计数与在线状态；
- 管理端通过触摸按钮（REQ DHT/REQ MPU）或硬件按键（KEY1→DHT、KEY2→MPU）手动触发数据请求；
- 管理端可下发LED控制帧远程点亮/熄灭采集前端板载LED。

### 1.3 性能指标

| 指标项 | 要求 |
|--------|------|
| RS485波特率 | 115200bps 8N1 |
| 管理端轮询周期 | 10s |

### 1.4 嵌入式设计约束

- **实时性约束**：FreeRTOS抢占式调度（1000Hz系统节拍），RS485接收任务优先级最高（6），确保115200bps速率下字节不丢失；
- **总线带宽约束**：轮询周期10s，单次通信数据量<30字节，总线占用率远低于1%，有充足裕量；
- **存储容错约束**：CRC-8校验（多项式0x07），校验范围覆盖帧头至数据末字节，校验失败直接丢弃不完整帧；
- **电源与电磁兼容**：RS485差分信号共模抑制比高，但PCB布线需注意DE/RE控制线与SDRAM数据线的串扰隔离。

**硬件选型分析：**
- **STM32F429IGTx**：Cortex-M4@180MHz，内置LTDC控制器驱动800x480 RGB屏，集成USART2/3用于RS485通信，2MB Flash + 256KB SRAM满足FreeRTOS多任务内存需求；
- **SP485E收发器**：3.3V供电，最高10Mbps，PB8控制DE/RE方向切换，发送前需≥1μs稳定延时；
- **DHT22温湿度传感器**：单总线协议，3.3-5.5V供电，精度±0.5°C/±2%RH，PG9数据引脚需4.7kΩ上拉；
- **MPU6050六轴IMU**：I2C1接口（PB6/PB7），400kHz快速模式，内置DMP运动处理器输出欧拉角；
- **GT9xx电容触摸屏**：I2C2接口（PH4/PH5），软件I2C驱动，PD11(RST)/PD13(INT)，支持5点触控；
- **W25Q128 SPI Flash**：16MB存储，SPI5接口，预留固件升级与大容量数据存储；
- **IS42S16400J SDRAM**：8MB@0xD0000000，LTDC帧缓冲 + GUI绘图缓冲。

本人负责管理端主机（地址0x01）的硬件搭建与软件实现。硬件连接关系如下：

- **MCU**：STM32F429IGTx，LQFP176封装，180MHz主频；
- **RS485通信**：USART2（PD5=TX, PD6=RX），SP485E收发器PB8控制DE/RE方向；
- **显示**：LTDC RGB565接口驱动800×480 TFT LCD，帧缓冲位于外部SDRAM地址0xD0000000；
- **触摸**：GT9xx电容触摸屏，I2C2（PH4/PH5）软件I2C，PD11(RST)/PD13(INT)；
- **按键**：KEY1(PA0)请求DHT、KEY2(PC13)请求MPU6050，外部下拉电阻，按下高电平；
- **LED**：3路RGB LED（PH10/PH11/PH12），低电平有效；
- **存储**：W25Q128 16MB SPI Flash（SPI5），预留固件升级空间。

**设计影响因素分析：**
1. RS485 DE控制引脚（PB8）与LTDC数据线（PB8/LTDC_B6）复用，需在LCD初始化中禁用PB8的LTDC功能（注释LTDC_B6 GPIO初始化），避免引脚冲突导致总线锁死；
2. SDRAM 180MHz高速写入时产生电磁辐射，PD6(RX)邻近SDRAM数据线，初期出现帧缓冲数据串扰至USART接收端的现象（表现为收到规律像素模式数据），通过DE切换时序优化（发送前清SR+DR，切RX后立即开RXNE中断）解决；
3. GT9xx触摸屏INT引脚（PD13）使用EXTI15_10中断，ISR中仅释放二值信号量（<5μs），I2C读取全部在GUI_Task任务上下文完成，避免ISR阻塞。

## 五、程序设计及原理分析

### 5.1 CRC-8校验算法

RS485总线帧采用CRC-8校验（多项式 x⁸+x²+x+1=0x07），生成256字节查找表。校验覆盖范围从帧头0xAA至CRC字节前一字节。

```c
static const uint8_t crc8_table[256] = {
    0x00,0x07,0x0E,0x09,0x1C,0x1B,0x12,0x15,
    /* ... 完整256字节查找表 ... */
};

uint8_t CRC8_Compute(const uint8_t *data, uint16_t len)
{
    uint8_t crc = 0;
    while (len--) {
        crc = crc8_table[crc ^ *data++];
    }
    return crc;
}
```

**原理：** 初始CRC值0x00，对每个输入字节执行 `crc = table[crc ^ byte]` 迭代。查表法相比逐位异或运算，速度提升约8倍，适合115200bps持续数据流的实时校验。

### 5.2 协议状态机实现

Protocol_FeedByte()实现7状态有限状态机：

```
STATE_WAIT_HEADER → 0xAA → STATE_WAIT_ADDR → 地址过滤 → STATE_WAIT_LEN
→ 长度检查 → STATE_WAIT_TYPE → STATE_WAIT_DATA → STATE_WAIT_CRC
→ CRC校验 → STATE_WAIT_TAIL → 0x55 → 帧就绪(frame_ready=1)
```

**地址过滤逻辑：** 
- 管理端 `is_manager=1` → 接收所有地址的帧（用于监控总线）
- 采集前端 `is_manager=0` → 仅当 `byte == local_addr` 时继续解析，否则丢弃复位

**CRC失败处理：** 当接收到CRC字节与计算值不匹配时，直接重置状态机至WAIT_HEADER并清零byte_count，丢弃整帧。管理端对CRC失败的帧不计入500ms超时判断，继续等待有效应答。

### 5.3 帧组包函数

```c
uint8_t Protocol_BuildResponse(uint8_t *out_buf, uint8_t dest_addr,
                                uint8_t msg_type, const uint8_t *data, uint8_t data_len)
{
    uint8_t idx = 0;
    uint8_t len = data_len + 1;           /* Len = Type+Data */

    out_buf[idx++] = PROTOCOL_HEADER;     /* 0xAA */
    out_buf[idx++] = dest_addr;
    out_buf[idx++] = len;
    out_buf[idx++] = msg_type;

    if (data_len > 0 && data != NULL) {
        memcpy(&out_buf[idx], data, data_len);
        idx += data_len;
    }

    out_buf[idx++] = CRC8_Compute(out_buf, idx);  /* CRC覆盖AA~数据末 */
    out_buf[idx++] = PROTOCOL_TAIL;               /* 0x55 */
    return idx;
}
```

### 5.4 RS485收发控制（DE方向切换）

SP485E收发器DE引脚（PB8）控制收发方向。发送时必须先禁用RXNE中断、拉高DE、延时稳定、逐字节发送、等待TC完成标志、拉低DE切回接收、立即重新使能RXNE中断。

```c
void RS485_Send_Data(uint8_t *buf, uint8_t len)
{
    USART_ITConfig(RS485_USART, USART_IT_RXNE, DISABLE);  // 关接收中断
    RS485_TX_EN();                                         // DE=H 发送模式
    for(volatile int i=0; i<1000; i++);                   // 稳定延时

    for(uint8_t t=0; t<len; t++) {
        while(USART_GetFlagStatus(USART2, USART_FLAG_TXE)==RESET);
        USART_SendData(USART2, buf[t]);
    }
    while(USART_GetFlagStatus(USART2, USART_FLAG_TC)==RESET);  // 等发送完成

    RS485_RX_EN();                           // DE=L 接收模式
    { volatile uint32_t s=USART2->SR; volatile uint32_t d=USART2->DR; (void)s;(void)d; }
    USART_ITConfig(RS485_USART, USART_IT_RXNE, ENABLE);  // 开接收中断
}
```

**关键设计要点：**
1. 发送前关闭RXNE → 防止SP485E内部DI→RO回环产生自收数据
2. DE切回后立即开RXNE → 确保采集端快速应答的起始字节不丢失
3. 读SR再读DR → 清除TX期间可能产生的ORE/FE等错误标志

### 5.6 MPU6050 DMP解算姿态角

采集前端2通过I2C1读取MPU6050的DMP（Digital Motion Processor）固件输出。DMP内部完成加速度计+陀螺仪的六轴传感器融合，直接输出欧拉角（pitch/roll/yaw），每秒200Hz数据更新率，比原始数据滤波更稳定。

```c
uint8_t MPU6050_DMP_GetData(float *pitch, float *roll, float *yaw)
{
    short gyro[4], accel[4], sensors;
    long quat[4];
    dmp_read_fifo(gyro, accel, quat, NULL, &sensors);
    if (sensors & INV_WXYZ_QUAT) {
        /* 四元数→欧拉角转换，InvenSense官方库自动处理 */
        *pitch = asin(-2*(quat[1]*quat[3]-quat[0]*quat[2])) * 57.2958f;
        *roll  = atan2(2*(quat[0]*quat[1]+quat[2]*quat[3]),
                       1-2*(quat[1]*quat[1]+quat[2]*quat[2])) * 57.2958f;
        *yaw   = atan2(2*(quat[0]*quat[3]+quat[1]*quat[2]),
                       1-2*(quat[2]*quat[2]+quat[3]*quat[3])) * 57.2958f;
        return 1;
    }
    return 0;
}
```

### 5.7 GUI触摸交互逻辑

GUI_Task每20ms周期循环，集成了按键扫描、触摸信号量检测、传感器数据队列排空、LCD全屏刷新。触摸按键使用400ms消抖窗口避免单次触摸触发两次请求。

### 5.8 历史数据存储

各终端采用环形缓冲区存储最近STORE_DEPTH（10组）传感器数据：

```c
typedef struct {
    SensorRecord_t records[STORE_DEPTH];
    uint8_t head;    /* 写入位置 */
    uint8_t count;   /* 已存储条数(≤STORE_DEPTH) */
} DataStore_t;

void DataStore_Push(DataStore_t *store, const SensorRecord_t *rec) {
    store->records[store->head] = *rec;
    store->head = (store->head + 1) % STORE_DEPTH;
    if (store->count < STORE_DEPTH) store->count++;
}
```

管理端分配两个DataStore实例分别保存前端1温湿度与前端2姿态角历史数据。

### 6.2 子模块测试结果

**（1）温湿度采集测试**

管理端通过RS485向采集前端1发送请求帧 `AA 02 01 01 A7 55`，前端返回 `AA 01 05 01 01 2C 01 F4 27 55`（温度30.0°C，湿度50.0%）。DHT22采集数据与标准温湿度计对比，实验室环境（24°C/55%RH）下误差在规格范围内。

**（2）RS485帧超时与重传测试**

管理端断开采集前端2的RS485线缆，总线仅剩高阻态。管理端按KEY2触发MPU6050请求，RS485TxTask发出请求帧后500ms内未收到有效应答帧，串口打印重试信息后放弃本次请求。重连前端2后通信自动恢复正常。

**（3）CRC异常容错测试**

使用Python监控台向管理端发送故意篡改CRC字节的温湿度帧 `AA 01 05 01 01 2C 01 F4 FF 55`（CRC错误）。管理端Protocol_FeedByte状态机检测CRC不匹配后立即重置至WAIT_HEADER，不触发`[RX] DHT`输出，LCD数据不更新。随后发送正确帧后系统立即恢复正常。

**（4）GUI触摸按钮测试**

触摸DHT按钮区域→打印`[TOUCH] DHT`→发送请求帧→收到应答→LCD温度湿度更新。400ms消抖窗口有效：连续快速点击两次，仅触发一次请求。

**（5）LED远程控制测试**

管理端RS485RxTask收到 `AA 01 02 03 01 D6 55`（LED ON）→ 调用LED1_ON → 板载LED1(红色)亮起。发送 `AA 01 02 03 00 3E 55`（LED OFF）→ LED1熄灭。响应延迟<10ms。

### 6.3 整体联调测试

三台设备全部上电并接入同一RS485总线，使用自研Python RS485实时监控台（rs485_monitor.py）监听总线通信。测试流程：
1. 管理端按KEY1→总线出现`AA 02 01 01`请求帧→前端1应答温湿度帧→管理端LCD左侧更新→监控台左侧大数字同步刷新；
2. 管理端按KEY2→同上流程，MPU6050姿态角更新；
3. 触摸DHT/MPU按钮→总线出现对应请求帧→前端应答→所有设备同步更新；
4. 断开前端1通信→管理端500ms超时打印重试信息→重连后恢复正常；
5. 管理端发送LED控制帧→前端LED实时响应。

## 八、总结

### 8.1 系统完成度

本项目完整实现了1台管理端+2台采集前端通过RS485总线的多节点通信。核心功能全部达标：温湿度采集（DHT22，5s）、姿态角采集（MPU6050 DMP，2s）、RS485帧协议（AA+Addr+Len+Type+Data+CRC+55）、总线超时容错（100ms帧超时+500ms应答重传）、LCD GUI实时显示与历史存储（最近10组）、LED远程控制。自研Python RS485监控台实现了总线数据的实时解析与大数字可视化。系统稳定运行超过30分钟无通信异常。

### 8.2 软硬件设计优缺点

**优点：**
- FreeRTOS抢占式多任务架构，RS485接收任务优先级最高，保证115200bps不丢字节；
- CRC-8校验+地址过滤+超时重传三重容错机制，通信可靠性高；
- 软件I2C触摸驱动采用ISR极轻量化设计（仅give信号量），I2C通信在任务上下文完成，避免中断阻塞；
- LCD GUI卡片式布局、实时数据刷新、历史趋势显示，用户体验良好。

**缺点与改进方向：**
- FreeRTOS堆空间仅36KB，GUI_Task栈深度大（8KB），多开任务时存在堆溢出风险，建议后期升级到64KB堆；
- 当前轮询周期固定10s，未实现动态轮询（根据数据变化量调整频率）；
- 采集前端传感器数据仅在本地保存10组环形缓冲，未使用SPI Flash做持久化存储；
- RS485总线未使用Modbus RTU等工业标准协议，自定义协议虽简单但缺乏扩展性。

### 8.3 FreeRTOS多任务调度优化点

1. 当前GUI_Task（pri 3）与SensorTask（pri 4）优先级差距较大，传感器数据更新后GUI最多延迟20ms才能刷新，可提升GUI_Task优先级至4以减少显示延迟；
2. RS485TxTask（pri 5）发送期间禁用RXNE中断，高频率按键（<50ms间隔）时可能丢失应答帧的第一字节，建议改用DMA+TC中断方式替代阻塞等待。
